# ReTrace-Bench v2: Memory Reliability in Realistic Workflows

ReTrace-Bench v2 evaluates agent memory reliability in realistic workflows. It is strictly evaluation-only and separate from ReTrace-Learn. It is not designed only for ReTrace-Learn typed revision actions.

## Motivation

Large Language Model (LLM) agents deployed in enterprise workflows interact with multiple systems, users, and tools over long horizons. Maintaining a reliable long-term memory is critical for task success. However, existing benchmarks primarily focus on static fact retrieval, simple chat history, or one-dimensional memory operations. They fail to capture complex multi-agent workflows, state evolution, implicit conflict resolution, or policy boundaries.

## Why v1 was too narrow

ReTrace v1 was heavily tailored around the ReTrace-Learn framework's specific DPA (Defeat-Path Authorization) kernel and typed revision vocabulary (`SUPERSEDES`, `BLOCKS`, etc.). While useful for validating the engine, it forced an implementation-biased framing on all benchmark participants. In realistic setups, a memory manager or agent should be evaluated on general memory reliability behaviors, regardless of whether it uses structured revision internally.

## v2 Design Principles

1. **Model-Agnostic / Implementation-Agnostic**: Any memory manager or LLM agent must be able to participate without supporting specific DPA graphs or typed actions.
2. **Realistic Industrial Workloads**: Scenarios model actual workflows (e.g. software development, customer service, calendar scheduling) with multi-agent traces, tool calls, and environment notifications.
3. **Hidden Memory Lifecycle**: Gold states and timelines are generated by deterministic workflows or simulators rather than subjective LLM-judged labels.
4. **Auditability and Grounding**: Predictions must be groundable in specific event or memory traces to ensure correct reasoning.

## Industrial Event-Trace Framing

Instead of simple dialogue histories, a scenario in v2 is represented as an **Event Trace**. Events model user utterances, agent actions, tool calls, memory reads/writes, and system-level logs. This structure allows evaluating how agents retrieve and synthesize memory context in response to changing environments.

## Protocols

To support different levels of diagnostic access, v2 defines four protocols:
- **Black-box Task Protocol (Main Track)**: The primary public track. Agents are given an event trace and asked to perform a task. No DPA or internal memory representation needs to be exposed.
- **Structured Revision Protocol (Optional Diagnostic Track)**: An optional track using the ReTrace typed revision vocabulary to diagnose how models propose memory updates under newly presented evidence.
- **Memory-State Protocol**: Evaluates the agent's ability to track the state of individual memories (e.g. active vs. blocked).
- **Oracle Diagnostic Protocol**: An internal track to calibrate baselines using gold revision paths.

## Contamination Policy

To ensure absolute clean methodology and avoid test-set leakage, `data/retrace_bench/` is strictly evaluation-only. Benchmark data must never be used for SFT or RL tuning of ReTrace-Learn or other method-centric models.

## Staged Dataset Plan

To maintain high data quality and avoid synthetic artifacts, the dataset scales in four phases:
1. **sample_20_v2**: A toy public sample of $\le 20$ scenarios to validate schemas and run smoke tests.
2. **seed_100**: A 100-scenario core evaluation set curated and audited entirely by humans.
3. **alpha_1000**: An intermediate 1000-scenario dataset.
4. **full_2500**: The final 2500-scenario evaluation benchmark.
